Here's a compact but complete PRD you can paste into Claude Code as the "master spec" for scaffolding this system over time.  ***  # Product Requirements Document (PRD)   **Product**: Atlas OS – Personal AI Hive-Mind Platform   **Owner**: Solo AI Engineer (you)   **Timeline**: ~12 months from first use to "v1 launch"  ***  ## 1. Vision & Goals  ### 1.1 Vision  Enable a single user to operate like an **Apple-scale AI platform team** by turning all personal devices into nodes of one coherent **AI hive mind**:  - One **world model** (Atlas OS) describing agents, services, workflows, and devices.   - One **control plane** to orchestrate AI work across devices.   - One **observability layer** that logs and analyzes all important AI interactions.    New hardware should join this hive mind as easily as "log in with Google" (or a simple script), contributing compute and capabilities to the unified system.  ### 1.2 Primary Goals  By v1 launch (1 year):  - A **single world-root repo** (the "Atlas world") defining:   - Environment configs, prompts, agents, orchestrators, policies.     - Device roles and their capabilities.   - A **per-device agent** that:   - Registers with the world.     - Pulls relevant config.     - Exposes local capabilities (filesystem, tools) via a standard interface.   - A **unified AI logging/tracing layer**:   - Captures key metadata for every important AI interaction (model, context, world state, device, outcome).[1][2][3] - Basic **context/decision graph**:   - Ability to query "what context was used when X happened?" and "which flows work best?"    ### 1.3 Non-Goals (v1)  - No full distributed OS or kernel replacement.   - No multi-user tenancy (single user only).   - No hard dependency on any specific vendor (Claude, DeepSeek, etc. pluggable).  ***  ## 2. Target Users & Use Cases  ### 2.1 Primary user  - A single **power-user / AI engineer** who:   - Works primarily via terminal and AI coding assistants.     - Has multiple devices (desktop, laptop, phone) and wants them to feel like one system.    ### 2.2 Core use cases  1. **Unified dev environment**      - Start work from any device and see the same world structure, agents, and workflows.      - Environment is reproducible on new hardware with minimal commands.  2. **Observable AI interactions**      - For any important AI call, see:      - Which device it came from.        - Which world-state commit was active.        - Which context (files/prompts) were used.        - Whether the outcome was good/bad.[4][5][6]  3. **Incremental scaling**      - Add a new node (e.g. NUC or cloud box) via login or script; it:      - Registers with Atlas.        - Pulls world config.        - Starts accepting tasks (e.g. long-running jobs, background indexing).  ***  ## 3. System Overview & Architecture  ### 3.1 High-level components  1. **Atlas World Repo ("World Model")**    - Git repo (initially on primary device) that defines:      - `atlas/config/` – world configuration (devices, roles, policies).        - `atlas/agents/` – agent definitions (prompts, capabilities, routing).        - `atlas/services/` – service definitions, orchestrator flows.        - `atlas/policies/` – safety, access rules, quotas.        - `atlas/logging/` – schemas for AI call logs and context events.        - `atlas/docs/` – human-readable docs generated from schemas.      - Backed by **chezmoi** for machine-specific configuration where appropriate (shell, editor, CLI tools).[7][8]  2. **Atlas Node Agent (per-device)**    - Lightweight daemon/CLI (e.g. Go or Python) that runs on each node:      - Reads local config.        - Registers device with Atlas World (via local or remote control API).        - Exposes capabilities:        - CPU/GPU info.          - Filesystem roots (read-only by default).          - Supported tools (e.g. `python`, `docker`, custom scripts).        - Applies relevant world config (e.g. via chezmoi on Linux/macOS).[8][7]  3. **Control Plane API (Atlas Core)**    - A small service (can start as a single process on primary device) that:      - Maintains a registry of devices, agents, services.        - Provides discovery and routing for commands:        - "Run this job on a capable device."          - "Fetch logs for this agent."        - Reads its configuration from the world repo and reloads on change.  4. **AI Logging & Context Tracing**    - Instrumentation/wrapper that:      - Intercepts calls to AI providers (Claude, DeepSeek, Gemini, etc.).        - Attaches meta        - `world_commit`, `world_branch`, `device_id`.          - `agent_id`, `tool`, `context_refs` (paths/IDs).          - `result`, `rating` if available.[2][3][9]      - Stores logs in append-only format (JSONL/DB) under `atlas/logs/` or external observability tool.  5. **Atlas OS Interface**    - CLI and/or TUI that:      - Presents a **human-readable map** of:        - Devices and their roles.          - Agents/services and their locations.          - Recent AI interactions and context flows.        - Provides basic commands:        - `atlas status`, `atlas devices`, `atlas agents`, `atlas logs tail`, etc.  ***  ## 4. Functional Requirements  ### 4.1 World Repo & Config  - FR-1: Define an **initial directory structure** and configuration schema for the Atlas world (YAML/JSON + Markdown).   - FR-2: Support **per-device overrides** using templates or device-specific config (via chezmoi or local overlays).[10][7] - FR-3: Allow the world repo to be cloned and applied on a new device with a small bootstrap script.  ### 4.2 Node Registration & Capabilities  - FR-4: Each node runs an Atlas agent that:   - Generates a **device ID** and registers with the control plane.     - Advertises capabilities (CPU/GPU, OS, network reachability, tags).   - FR-5: Node agent periodically heartbeats; Atlas Core marks nodes as online/offline.  ### 4.3 Task Routing & Execution (v1 minimal)  - FR-6: Provide a simple **task execution API**:   - Submit a job with:     - desired capabilities (e.g. GPU, OS).       - command or script ref.     - Atlas selects an appropriate node and runs it via the node agent. - FR-7: Support at least:   - "run this shell command"     - "run this script from the world repo"     - Future: "run this AI pipeline definition"  ### 4.4 AI Logging & Tracing  - FR-8: Provide a thin wrapper around AI providers that:   - Logs calls with:     - provider, model, input metadata, output summary.       - world-state identifiers (`world_commit`, `branch`).       - device, agent, tool identifiers.[3][2][4] - FR-9: Store logs in a structured, queryable format:   - Start with JSONL under `atlas/logs/`.     - Plan for integration with an observability tool (e.g. Langfuse or similar) later.[9][11][2]  ### 4.5 Atlas OS Interface  - FR-10: CLI commands (examples):   - `atlas init` – initialize world repo structure.     - `atlas status` – show devices, world branch/commit, recent events.     - `atlas devices` – list nodes and capabilities.     - `atlas logs tail` – stream AI interaction logs.   - FR-11: Optional TUI/dashboard (later phase) showing:   - Topology (world → devices → agents/services).     - Recent AI calls and context usage.  ***  ## 5. Non-Functional Requirements  - NFR-1: **Single-user first**; safe by default, but no multi-tenant complexity.   - NFR-2: Maximize use of **standard protocols** (SSH, HTTPS, Git, etc.), avoid custom networking where possible.   - NFR-3: **Incremental deployability**:   - Each component is useful on its own (world repo alone, logging alone, etc.).   - NFR-4: **Low overhead** on devices; node agent should be lightweight.  ***  ## 6. Phased Delivery Plan  ### Phase 0 (Month 0–1): Local World & Observability  - Deliverables:   - Atlas world repo locally on primary box.     - Basic directory structure (`atlas/config`, `atlas/agents`, `atlas/docs`, `atlas/logs`).     - Simple AI logging wrapper capturing world-state + device metadata.   - Success:   - You can see logs of your AI work with world commit + context refs from one machine.  ### Phase 1 (Month 1–3): Node Agent & Control Plane (Single Machine)  - Deliverables:   - Node agent running on primary box.     - Atlas Core service managing a registry (even if only one node).     - CLI to see `atlas status`, `atlas devices`.   - Success:   - Single-node but structured: you can "register" your main machine and see it through Atlas.  ### Phase 2 (Month 3–6): Multi-Device Federation  - Deliverables:   - Node agent runs on laptop.     - Control plane sees both machines.     - World repo and minimal per-device config sync (e.g., via Git + chezmoi).   - Success:   - Both machines are visible; tasks/commands can be routed to a specific node.  ### Phase 3 (Month 6–9): Unified Workflow & Context Graph  - Deliverables:   - Stronger AI logging across devices.     - Basic querying of logs to produce **context/decision graphs** (using scripts or a simple graph DB).     - Atlas CLI/TUI to navigate world → devices → agents → logs.   - Success:   - You can answer: "What context was used for this decision?" and "Which flows work best?" across devices.  ### Phase 4 (Month 9–12): Hardening & "v1 Launch"  - Deliverables:   - Hardening of security, key management, and error handling.     - Documentation and scripts to bootstrap a new device into the hive via a few commands.     - Optional packaging (install scripts, containers) for reuse or sharing.   - Success:   - You can reset, add, or replace hardware and rejoin it to Atlas quickly; the system is robust enough for daily use.  ***  ## 7. Risks & Mitigations (for Claude to keep in mind)  - **Complexity creep**: keep each phase small and usable on its own.   - **Partial observability**: unify a few main entrypoints for AI calls, don't chase 100%.   - **Overfitting to current vendors**: log with a generic schema; treat providers as plugins.   - **Single point of failure (you)**: design defaults and scripts so that "the easy way is the correct way."  ***  ## 8. What Claude Code should do with this PRD  - Ask clarifying questions about:   - Language/runtime preferences (Go vs Python for agents, etc.).     - Storage choices for logs (files vs DB).   - Then progressively scaffold:   - World repo structure and config schemas.     - Node agent skeleton and control plane API.     - Logging wrapper and initial CLI/TUI.     - Scripts to bootstrap devices and keep world state in sync.  Use this PRD as the single source of truth for design decisions, keeping everything modular so that each phase is shippable and useful on its own.
